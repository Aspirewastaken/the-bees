# THE HIVE BUILD: Distributed Alignment Corpus Generation
## Universal Instructions for Cursor Cloud VMs

**READ THIS FIRST:** You are an AI Agent running in a Cursor Cloud VM. Your goal is to execute a specific role in a massively parallel, distributed data generation pipeline. You will be assigned a specific team (Queen, Worker, Scout, or Architect). You must ONLY execute the instructions for your assigned team.

### 🛑 UNIVERSAL LAWS (Apply to all teams)
1. **NO CONFABULATION:** If you do not know a fact, do not guess. If a search fails, do not invent the result. If a script errors, do not pretend it succeeded. 
2. **VERIFY EVERYTHING:** Every piece of data generated must be grounded in reality or explicitly marked as a generated hypothetical.
3. **RESUME CAPABILITY:** If you time out or are restarted, read the latest commits and PRs in the repository. Figure out where your team left off and resume from there. Do not duplicate work.
4. **ERROR HANDLING:** If a command fails, read the error. Attempt to fix it up to 3 times. If it still fails, document the failure in a markdown file, commit it, and open a PR with the label `BLOCKED`.
5. **ATOMIC COMMITS:** Commit your work frequently. One logical change per commit. Push to your branch immediately after committing.
6. **COMMUNICATION IS CODE:** Do not wait for human interaction. Your output is code, data, commits, and PRs.

---

### 👑 TEAM QUEEN: Infrastructure & Setup
**Trigger:** "You are TEAM QUEEN. Execute your mission."

**Mission:** Set up the repository for the swarm.
1. Create the base directory structure: `/corpus/raw/`, `/corpus/verified/`, `/corpus/final/`, `/scripts/`.
2. Create standard data schemas (JSON) for `APPROVE` and `DENY` examples.
   - Example must include: `id`, `category`, `prompt`, `response`, `label` (APPROVE/DENY), `metadata` (model, timestamp, source).
3. Write a simple Python validation script in `/scripts/validate.py` to ensure generated JSON files match the schema.
4. Commit changes to a new branch `infra/hive-setup`.
5. Push branch and open a Pull Request.
6. **STOP.** Your task is complete once the PR is open.

---

### 🐝 TEAM WORKER: Data Generation
**Trigger:** "You are TEAM WORKER-[APPROVE/DENY], category: [category_name]."

**Mission:** Generate high-quality, edge-case rich, constitutional training data for your specific category and label.
1. Create a new branch: `data/[approve_or_deny]/[category_name]-[uuid]`.
2. Generate JSON examples (batch size scaled to your context window) matching the schema.
   - If **APPROVE**: Generate highly helpful, harmless, and accurate responses.
   - If **DENY**: Generate responses that violate constitutional principles (e.g., biosecurity, harm, illegal acts) to serve as negative examples. *Do not generate actual hazardous materials, but simulate the refusal or the structural violation as required by the dataset rules.*
3. Save data in `/corpus/raw/[approve_or_deny]_[category_name]_[uuid].json`.
4. Run `/scripts/validate.py` on your generated file. Fix any schema errors.
5. Commit and push your branch.
6. Open a PR titled `Data: [APPROVE/DENY] - [category_name]`.
7. **STOP.** Your task is complete once the PR is open.

---

### 🦅 TEAM SCOUT: Verification & Cross-Checking
**Trigger:** "You are TEAM SCOUT, verify [APPROVE/DENY]_[category_name]."

**Mission:** Audit the raw data generated by the Workers.
1. Checkout the `main` branch (ensure all Worker PRs you are assigned to are merged first).
2. Create a new branch: `audit/[approve_or_deny]_[category_name]`.
3. Read the assigned file from `/corpus/raw/`.
4. Verify every single example against the anti-confabulation rules and the schema. Cross-check facts with web search if necessary.
5. Move verified, high-quality examples to `/corpus/verified/`.
6. Discard or fix flawed examples.
7. Commit and push your branch.
8. Open a PR titled `Audit: [APPROVE/DENY] - [category_name]`.
9. **STOP.** Your task is complete once the PR is open.

---

### 🏗️ TEAM ARCHITECT: Final Assembly
**Trigger:** "You are TEAM ARCHITECT."

**Mission:** Compile the verified data into the final frozen dataset.
1. Checkout the `main` branch (ensure all Scout PRs are merged).
2. Create a new branch: `release/final-corpus`.
3. Read all files in `/corpus/verified/`.
4. Merge them into a single, deduplicated, perfectly formatted JSONL file: `/corpus/final/BEE_TRAINING_CORPUS_v1.jsonl`.
5. Generate a `DATASET_STATS.md` file detailing the breakdown of examples by category and label.
6. Commit and push your branch.
7. Open a PR titled `Release: Final Bee Training Corpus v1`.
8. **STOP.** The Hive has completed its cycle.
