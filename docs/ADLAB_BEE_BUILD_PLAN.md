# ADLAB BEE BUILD PLAN
## Parallel Cursor Cloud execution plan (v1.0)

This is the operator-level plan for producing the Bee training corpus quickly and reproducibly.
The detailed role instructions live in `docs/THE_HIVE_BUILD.md`.

---

## 1) Mission

Deliver a **10,000-example verified binary corpus** for Bee classifier training:

- 5,000 APPROVE
- 5,000 DENY
- category-balanced
- scout-verified
- provenance-preserving

---

## 2) Inputs and outputs

### Inputs
- `docs/THE_BEE_SPEC.md`
- `docs/THE_HIVE_BUILD.md`
- `scripts/hive_cli.py`
- Category manifests generated by Team Queen

### Outputs
- `hive/final/bee_corpus_v1.jsonl`
- `hive/final/bee_corpus_v1_summary.json`
- `hive/final/DATASET_CARD.md`
- `hive/rejected/*` and human-review queue

---

## 3) Parallel execution waves

Before Wave A:

```bash
python3 scripts/hive_cli.py preflight --root .
python3 scripts/hive_cli.py make-manifests --root . --run-id <RUN_ID> --records-per-worker 500 --overwrite
```

## Wave A (Setup): 1 VM
**Team:** Queen  
**Goal:** create repo structure, schemas, manifests, prompts, quality gates.

Exit criteria:
- manifests validated
- schema checks pass
- Queen PR merged

## Wave B (Generation): 20 VMs in parallel
**Teams:** 10 Worker-DENY + 10 Worker-APPROVE  
**Goal:** produce 500 records per VM (10,000 total).

Exit criteria:
- each shard committed
- worker summaries present
- no manifest collisions

## Wave C (Verification): 20 VMs in parallel
**Team:** Scout  
**Goal:** independently verify Worker shards.

Exit criteria:
- every record receives scout verdict
- rejection and rework queues documented

## Wave D (Assembly): 1 VM
**Team:** Architect  
**Goal:** merge approved records, dedupe, balance, publish final corpus.

Exit criteria:
- final corpus + summary + dataset card committed
- unresolved cases listed for human review

---

## 4) Time and cost envelope (target)

- Setup: ~30-45 min
- Generation: ~3-4 hours (parallel)
- Verification: ~2-3 hours (parallel)
- Assembly: ~30-60 min

Total runtime target: **~8 hours** end-to-end.

Estimated cloud + API spend target: **~$300** (varies with model choice and retries).

---

## 5) Quality bar

Minimum quality requirements:

1. Schema-valid JSONL: 100%
2. Scout coverage: 100%
3. Critical policy violations in final set: 0 tolerated
4. Category and class balance documented
5. Full provenance links (worker shard + scout verdict)

---

## 6) Governance and anti-confabulation

Non-negotiables:

- No fabricated evidence, metrics, or citations
- Unknown facts marked explicitly as `UNKNOWN` or `UNVERIFIED`
- No silent post-scout edits; changed records require new IDs
- No final merge without scout evidence

When in doubt, fail closed and route to human review.

---

## 7) Operator checklist

1. Launch Queen VM and merge Queen PR.
2. Launch 20 Worker VMs with unique category assignments.
3. Merge Worker PRs.
4. Launch 20 Scout VMs on produced shards.
5. Merge Scout PRs.
6. Launch Architect VM and merge final assembly PR.
7. Archive release artifacts and snapshot manifests.

---

## 8) Risks and mitigations

### Risk: inconsistent labels across workers
Mitigation: strict schema, category manifest, and scout rejection path.

### Risk: duplicated or near-duplicate records
Mitigation: architect dedupe pass with exact + near-duplicate checks.

### Risk: confabulated references or fake metrics
Mitigation: anti-confab rules in all prompts and scout verification of claims.

### Risk: interrupted VM runs
Mitigation: checkpointed status files and idempotent shard writing.

---

## 9) Handoff

If this plan is handed to a new operator, they need only:

1. this file (`ADLAB_BEE_BUILD_PLAN.md`)
2. `THE_HIVE_BUILD.md`
3. repository access

The run is designed to be resumable at any wave boundary.
